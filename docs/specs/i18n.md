# Feature: Internationalization (i18n) Support

## Summary
Implement bilingual support (English and Indonesian) for the portfolio website. The language toggle UI already exists in Navbar (EN/ID buttons) but is non-functional. This feature makes the toggle operational with full content translation, locale-based URL routing, and persistent language preference.

## User Story
As a portfolio visitor, I want to view the site in my preferred language (English or Indonesian), so that I can understand the content in the language I'm most comfortable with.

## Acceptance Criteria

### AC1: Default Language Detection
- [ ] Given a first-time visitor with browser language `id` or `id-ID`, when they load the site, then the site displays in Indonesian with URL path `/id/`
- [ ] Given a first-time visitor with browser language other than `id`/`id-ID`, when they load the site, then the site displays in English with URL path `/en/`
- [ ] Given a returning visitor with saved language preference in localStorage, when they load the site, then the site displays in their saved language regardless of browser language

### AC2: Language Toggle Behavior
- [ ] Given the user is viewing any page, when they click the "EN" button in Navbar, then the site switches to English, URL changes to `/en/[current-route]`, and all translatable content re-renders in English
- [ ] Given the user is viewing any page, when they click the "ID" button in Navbar, then the site switches to Indonesian, URL changes to `/id/[current-route]`, and all translatable content re-renders in Indonesian
- [ ] Given the user switches language, when the content re-renders, then the scroll position is preserved at the same content location
- [ ] Given the user switches language, when the URL changes, then the language preference is persisted to localStorage key `portfolio_language`

### AC3: Visual Toggle State
- [ ] Given the current language is English, when viewing the Navbar, then the "EN" button appears visually highlighted/active and "ID" appears inactive
- [ ] Given the current language is Indonesian, when viewing the Navbar, then the "ID" button appears visually highlighted/active and "EN" appears inactive

### AC4: URL Structure and Routing

- [ ] Given the user navigates to `/id/`, when the page loads, then the site displays in Indonesian with all Indonesian content
- [ ] Given the user is on `/en/blog/debugging-rabbitmq-race-conditions`, when they switch to Indonesian, then the URL changes to `/id/blog/debugging-rabbitmq-race-conditions`
- [ ] Given the user directly navigates to a `/id/` path, when the page loads, then the site displays in Indonesian without redirect
- [ ] Given the user shares a URL with locale prefix (e.g., `/id/projects`), when another user opens it, then the site displays in that locale

## Scope: In / Out

**In Scope:**
- Bilingual support: English (EN) and Indonesian (ID)
- Language toggle UI functionality (EN/ID buttons in Navbar)
- Translation of all user-facing content per AC5-AC17
- URL-based locale routing (`/en/`, `/id/`)
- Browser language detection with English fallback
- localStorage persistence of language preference
- Date localization (Indonesian month abbreviations)
- Fallback to English for missing translations
- SkyForceGame UI text translation (menus, messages, buttons)

**Out of Scope:**
- Additional languages beyond EN and ID
- Translation of blog article content (titles/excerpts only)
- Translation of technical status labels, tech stack names, skill tags
- Translation of personal name and email address
- Translation of decorative code-style labels (per stakeholder decision)
- Social media placeholder links (per L-024 — documented future work)
- Right-to-left (RTL) language support
- Locale-specific number formatting (decimal separators, thousand separators)
- Locale-specific currency formatting
- Time zone localization

## Edge Cases

### Edge Case 1: URL Mismatch with localStorage
**Scenario:** User has Indonesian saved in localStorage but directly navigates to `/en/about`
**Expected Behavior:** URL takes precedence. Site displays in English. localStorage is updated to `en`.

### Edge Case 2: Invalid Locale in URL
**Scenario:** User navigates to `/fr/about` (unsupported locale)
**Expected Behavior:** Redirect to `/en/about` (default language). No error thrown.

### Edge Case 3: Language Switch During Form Input
**Scenario:** User is typing in Contact form, then switches language
**Expected Behavior:** Form input values are preserved. Only labels and placeholders change language. User input is not cleared.

### Edge Case 4: Language Switch During Game Play
**Scenario:** User is playing SkyForceGame, then switches language
**Expected Behavior:** Game continues without interruption. UI labels update to new language. Game state (score, lives, etc.) is preserved.

### Edge Case 5: Missing Translation for New Content
**Scenario:** Developer adds new component with text but forgets to add Indonesian translation
**Expected Behavior:** English text displays for Indonesian locale (per AC19). No error thrown. UI remains functional.

### Edge Case 6: Browser Language Detection Edge Cases
**Scenario:** User's browser language is `id-ID` (Indonesia), `id` (Indonesian), or `ms-MY` (Malay)
**Expected Behavior:**
- `id-ID` or `id` → default to Indonesian
- `ms-MY` or other → default to English (only `id` and `id-ID` trigger Indonesian default)

### Edge Case 7: Deep Link with Locale
**Scenario:** User receives shared link `/id/blog/prisma-orm-performance` via WhatsApp
**Expected Behavior:** Site loads in Indonesian with blog detail page displayed. No redirect to `/en/`. localStorage is set to `id`.

### Edge Case 8: SEO Crawlers
**Scenario:** Google bot crawls `/en/` and `/id/` paths
**Expected Behavior:** Both locales render correctly server-side or on initial load. No redirect loop. `hreflang` meta tags indicate alternate language versions (implementation detail for technical-architect).

## Technical Constraints

1. **No Server-Side Rendering (SSR):** Portfolio is a client-side SPA. Locale detection and routing must work entirely in browser.
2. **No Backend API:** All translations are bundled with frontend build. Translation files must be static JSON or TypeScript objects.
3. **React Context:** Language state must be managed via React Context (existing pattern in codebase — see ThemeContext).
4. **React Router DOM 7.14.2:** URL-based routing must use React Router DOM. Locale prefix must be part of route paths.
5. **Vite Build:** Translation files must be compatible with Vite build process. No dynamic imports that break build.
6. **localStorage Only:** Persistence uses localStorage. No cookies, no session storage.
7. **Performance:** Language switch must be instant (<100ms). No full page reload required.
8. **Existing Component Structure:** Translation implementation must not require full component rewrites. Minimal changes to existing components.
9. **Future Dynamic Content:** Translation structure must support future migration to CMS or API-loaded content (per AC21).

## Notes

- Language toggle UI already exists in Navbar.tsx (lines 12, 21, 102-108, 123-128) but only manages local state. Implementation must connect this UI to actual i18n functionality.
- Current codebase has no i18n library installed. technical-architect will select library and document in ADR.
- Translation source: Developer generates translations (machine translation + review acceptable per stakeholder decision).
- Blog posts will remain English in article body but metadata (title/excerpt) will be translated. This allows technical content to remain accurate while improving discoverability for Indonesian visitors.
- Decorative tech-style labels (e.g., "Distributed_Eng", "Security_First") remain English per stakeholder aesthetic decision.
- SkyForceGame high score localStorage keys (`skyforce_highscore`, `skyforce_highscore_name`) are not localized — they remain shared across locales so high scores persist when switching language.
